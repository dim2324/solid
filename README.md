# Принципы SOLID в архитектуре приложения

Данный проект демонстрирует применение принципов SOLID при разработке гибкой и поддерживаемой архитектуры.

---

## 🧱 Применённые принципы SOLID

### 1. Single Responsibility Principle (SRP)
> У каждого класса должна быть только одна причина для изменения.

- **`SQLiteOrderRepository`** — отвечает **только** за работу с базой данных  
- **`OrderService`** — отвечает **только** за бизнес-логику заказов  
- **`EmailSender` / `SMSSender`** — каждый отвечает **только** за свой тип уведомлений

---

### 2. Open/Closed Principle (OCP)
> Программные сущности должны быть открыты для расширения, но закрыты для модификации.

- Система **открыта для расширения** — можно добавлять новые типы БД и уведомлений  
- **Закрыта для модификации** — добавление нового функционала не требует изменения существующего кода  
- **`MultiNotifier`** — демонстрирует расширение без изменения существующих классов

---

### 3. Liskov Substitution Principle (LSP)
> Объекты в программе должны быть заменяемы экземплярами их подтипов без изменения корректности программы.

- Все реализации интерфейса `Notifier` **взаимозаменяемы**  
- `EmailSender`, `SMSSender`, `MultiNotifier` могут использоваться везде, где ожидается `Notifier`

---

### 4. Interface Segregation Principle (ISP)
> Клиенты не должны зависеть от интерфейсов, которые они не используют.

- `RepositoryInitializer` и `RepositoryWriter` — **разделены** на отдельные интерфейсы  
- Клиенты зависят **только** от тех методов, которые им действительно нужны

---

### 5. Dependency Inversion Principle (DIP)
> Зависимости должны строиться на абстракциях, а не на конкретных реализациях.

- `OrderService` зависит от **абстракций** (`RepositoryWriter`, `Notifier`), а не от конкретных классов  
- Все зависимости внедряются **через конструктор** (Dependency Injection)

---

## ✅ Преимущества новой архитектуры



| Преимущество | Описание |
|--------------|----------|
| 🚀 **Расширяемость** | Легко добавить новые БД (`PostgreSQL`, `MongoDB`) или уведомления (`Push`, `Telegram`) |
| 🧪 **Тестируемость** | Просто заменять зависимости на моки для unit-тестов |
| 🔄 **Гибкость** | Можно комбинировать разные реализации во время выполнения |
| 🛠️ **Поддерживаемость** | Чёткое разделение ответственности упрощает поддержку кода |
